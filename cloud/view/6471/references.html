Digitised Print Materials Metadata

  • Favourite
  • 1028 views

This cloud summarises the development of an application profile for use with digitised print objects in the OUDL

Cloud created by:

Alex Addyman
5 September 2012

 

The Open University Archives has digitised a range of print study materials which formed part of the Open University S100 Science Foundation Course during its early years. These are the text books which were sent out to students in order to guide them through the course. Each textbook has been digitised into PDFA and TIFF documents – one per original page.

 

Sources of legacy metadata

The S100 study materials were all originally catalogued into MARC21 format which is contained in the OU's Voyager Library System. This metadata was exported as MARCXML which enabled us to look at the range of MARC fields used and how they were used which is vital in creating a supportive application profile. The source metadata is fundamental to the OUDL as it partly shapes how we present our items and also imposes limitations on what we can do with them.

An example of the format of the export is shown below. To anyone unfamiliar with MARC or its XML version it is not particularly user friendly and won't display brilliantly well in our user interface. Also given that MARC is used to describe analogue bibliographic resources it isn't sufficient in describing compound digital objects with their greater complexity. We therefore need to map our metadata to a standard which is more suitable.

Selection of metadata standards

MODS, MARC, and Dublin Core

The OUDL Standards Review recommended that “where content already exists in MARC format, MARCXML or MODS should be used to allow the content to be expressed within the OUDL.” As we have already ruled out MARCXML we mapped our legacy metadata to MODS as shown in Figure 1. Dublin Core is also included in the mapping as this is a core requirement of OAI-PMH and is automatically generated for every XML object in Fedora.

MODS is derived from MARC-21 and so it is no surprise that it is both granular and detailed enough to map to every MARC field from our legacy metadata. Dublin Core by comparison does not offer the complexity to fully represent our legacy metadata but does offer basic metadata to support search functionality in Fedora. 

We then transformed the Voyager Catalogue data from MARCXML to MODS using the Library of Congress MARC to MODS Conversion Stylesheet. The transformation retained the descriptive and holdings delineation present in the source metadata. This is useful from an organisational perspective as we would wish to delineate the descriptive metadata from the analogue and gitial manifestation metadata inline with FRBR.

We will also wish to include metadata which is specific to the digital item and which captures the digitisation process. For guidance we can refer to theLibrary of Congress Metadata for Digital Content Master Element List version 4.1. This application profile sets out a list of core MODS elements to use in the description of digital objects.

 

Finalised element set

The finalised element set is shown in  Figure 2 and contains a series of elements derived from the legacy mapping exercise combined with some elements from the Library of Congress Metadata for Digital Content Master Element List version 4.1. The profile takes the metadata fields derived from the transformation MODS elements and organises them to show the separation between description and holdings. All MODS top-level elements and sub-elements remain true to their Library of Congress definition.

The profile also contains technical fields which will be harnessed from the digital objects. Some of these fields do not map to MODS elements and and so new elements have been created in their place. These fields will also be important in terms of Preservation and directly map to PREMIS elements (see Preservation Metadata Cloud).
