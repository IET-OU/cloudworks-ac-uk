Development of the OUDL collections-level metadata application profile

Cloud created by:

Alex Addyman
6 September 2012

The large and varied content which will make up OUDL requires a clear and easy-to-navigate hierarchical structure. To achieve this collections and sub-collections will be developed to group similar content together.

Collections Characteristics

As the development of the OUDL is an iterative process and as digital items suitable for inclusion into the repository will no doubt grow with time there are no fixed collections as yet. However the following collections themes have been identified whose titles and contents are subject to change.

OU Study Materials Archive 

Contents:

  • All the study materials (also called learning materials/course materials/teaching materials) in all formats. "Study Materials" was the preferred phrase when discussed with the Learning and Teaching team a while ago and is now the official title of the collection on the website etc.

OU Life Collection 

Contents:

  • OU Historical Images (non-teaching - this could be complicated as our images are jumbled together at the moment - teaching and historical)
  • OU Historical broadcasting (Open Forum TV and radio, other OU "magazine" programmes)
  • OU Vice-Chancellor's speeches (need to talk to the VC's office about these)
  • Other collections such as Sesame, Open House, OU web archive of social/comms stuff could also go into here.

OU Learning Journey Collection 

This collection title would be consistent with the OU/Agreement and the title now given to this material by OMU

Contents:

  • All fully-funded and co-produced non-teaching content - e.g. Coast etc

Legacy metadata

Collections-level metadata for digital content at the OUDL is non-existent. The closest thing we can draw on is the EAD encoded archival descriptions metadata used to describe physical archives collections such as the Jennie Lee Collection and the Walter Perry Collection. These are important to map because they contain useful identifier metadata and because there is the intention to preserve these records in the OUDL in the future.

Standards

The standards recommended in the OUDL Standards Review [1] for collections metadata are the Dublin Core Collections Application Profile and Encoded Archival Description. As well as these MODS has been explored as it has been used for other content types and has been profiled for collections in other scenarios [2].

Dublin Core Collections-level Application Profile

The Dublin Core Collections Application Profile is one of the most widely used collections level profiles. Because it draws on Dublin Core standards it is highly interoperable with other schemas and libraries. As is the case with the simple DC schema however it is somewhat limited in scope but at the collections level this is less of an issue. DC-CAP draws on the functional model and element set defined by RSLP.

DC-CAP can describe both the collection itself or a catalogue or index of the collection. It has been decided by the OUDL Project Team that the profile should describe the collection itself to act as a ‘top-layer’ for item and object level metadata.

Encoded Archival Description

Because many of the collections being described have an archival or historic element our application profile must either incorporate or map to Encoded Archival Description.[3] EAD is also a good schema to draw on because its natural structure which is derived from the International Standards for Archival Description[4] reflects the hierarchical nature of collections. EAD is used by the OU Archive to describe current physical collections and will provide metadata which can be transferred to the OUDL.

Selecting the elements

We determined a list of desirable elements drawn from the OU Archives Collections legacy metadata (in EAD) and from examples drawn from a literature review detailed in the appendices. This list was then cross-walked against Dublin Core Simple, DCCAP and MODS as shown below.

Simple Dublin Core was not sufficient to describe our collections (but will be able to describe them minimally for our OAI-PMH requirements). With the addition of qualifier terms through DCCAP was able to cover more of the required fields but with shortcomings around some of the ostensibly historic fields. For example ‘biographical history’ ( in EAD) does not map to a DCCAP field. On consulting another attempt[i] to crosswalk EAD to DCCAP this field has caused difficulty.

These cross-walking issues expose the need for two profiles. The first based on DCCAP will be used to describe new collections. DCCAP has sufficient depth to describe such collections but not enough depth or granularity to describe the archival collections. Where and when there is a need to incorporate archival collections in OUDL EAD will be used. This, we know from the cross-walk, can be mapped to MODS and DC and so will be extensible enough for our requirements.
